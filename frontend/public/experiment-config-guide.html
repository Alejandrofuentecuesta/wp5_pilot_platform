Como esta montado un experimento en WP5

Esta guia resume la configuracion real del repositorio: que define un experimento, como entra un participante, como corre el pipeline STAGE, que prompts se usan por defecto y que cosas puedes cambiar desde wizard, API o solo tocando codigo.

Director Update + Evaluate + Action Performer genera el mensaje Moderator limpia salida Classifier etiqueta incivility + like-mindedness

Lo mas importante

La config no es realmente inmutable hoy. El backend expone PUT /admin/config/{experiment_id}, asi que un experimento ya guardado se puede editar.
El prompt configurable de verdad es el classifier user template. Existe en tipos/backend, pero el wizard actual no lo muestra.
El display name del humano no viaja al backend. En runtime el backend solo conoce a la persona como participant.
Hay prompts legacy en prompts/system y prompts/user. El runtime actual usa sobre todo los templates unificados y los tres prompts del Director.

Mapa rapido

Un experimento guarda dos blobs JSON en la tabla experiments: simulation y experimental. Cada sesion carga ambos, pero el bloque experimental.groups[treatment_group] decide el tratamiento concreto de cada participante.

1
Admin
Define el experimento

ID, descripcion, ventana temporal, LLMs, grupos, features, tokens.

2
Token
Consume un acceso de un solo uso

El token fija a que treatment_group entra la sesion.

3
Session
SimulationSession crea el runtime

Carga config desde DB, inicializa agentes, features, LLM managers y clock loop.

4
Stage
El orquestador decide y ejecuta

Update, Evaluate, Action, Performer, Moderator y luego Classifier.

5
Output
Se persiste y se emite por Redis/WebSocket

Mensajes, likes, eventos, bloqueos y labels quedan en DB.

Estimador rapido

El loop corre cada segundo. La probabilidad de disparar un turno es messages_per_minute / 60. El valor siguiente es una expectativa teorica, no una garantia, porque la latencia del pipeline y los features tambien frenan.

30 turnos esperados

Formula simple: duracion * messages_per_minute.

10% probabilidad por tick

Cada segundo el sistema decide si intenta un turno nuevo.

5 defaults actuales del wizard

Duracion 5 min, 5 agentes, semilla 42, ventana de accion 5.

Configuracion persistida

  • experiments.config.simulation guarda pacing, nombres, modelos y sampling.
  • experiments.config.experimental guarda contexto, criterios y grupos.
  • tokens asocia cada codigo a un grupo experimental.
  • sessions congela la config usada al arrancar la sesion.

Privacidad y anonimato

  • El nombre visible del humano vive en localStorage del navegador.
  • Backend usa siempre participant como remitente real.
  • LLM ve alias anonimizados tipo Performer 1, Performer 2.
  • La anonimizacion usa random_seed para que el shuffle sea estable por sesion.

Features por grupo

  • news_article: siembra un mensaje [news] al inicio.
  • gate_until_user_post: silencia agentes hasta que el participante escriba.
  • Las features se definen por grupo, no globalmente.
  • Compat legacy: el viejo escenario news_article equivale a ambas features.

Wizard y defaults

Los valores iniciales salen del frontend, pero el backend vuelve a validar y en algunos casos aplica fallbacks si faltan claves. Eso significa que el default visible en el wizard y el fallback de API no siempre coinciden.

Step 1. Experiment Identity scope: experimento

Define el contenedor del experimento: experiment_id, description, starts_at, ends_at y redirect_url.

ID sin default Ejemplo: Bloque_1_Alejandro Descripcion obligatoria Start default = ahora End default = +24h Redirect default = vacio
  • Si redirect_url esta vacio, el frontend muestra la thank-you page interna.
  • La ventana temporal afecta al consumo de tokens antes de arrancar sesion.
  • Al editar un experimento, el ID queda bloqueado en UI.
Step 2. Session & Agents scope: simulation
duration = 5 random_seed = 42 num_agents = 5 agent_names = ["", "", "", "", ""] messages_per_minute = 6 evaluate_interval = 5 action_window_size = 5 performer_memory_size = 3
  • agent_personas es opcional y arranca vacio; solo aparece si lo activas.
  • Los nombres son obligatorios antes de guardar, pero el wizard empieza con placeholders vacios.
  • evaluate_interval no solo marca cadencia; tambien fija la longitud del contexto para Evaluate.
  • El backend aceptaria action_window_size = 10 como fallback si la API no lo manda, aunque el wizard mande 5.
Step 3. LLM Pipeline scope: por rol
Director: anthropic / claude-haiku-4-5 / 0.8 / 0.8 / 1024 Performer: mistral / mistral-large-latest / 0.8 / 0.8 / 256 Moderator: anthropic / claude-haiku-4-5 / 0.2 / 1.0 / 256 Classifier: anthropic / claude-haiku-4-5 / 0.2 / 1.0 / 256
  • El wizard deja copiar Director -> Performer y Moderator -> Classifier.
  • El test LLM usa el prompt fijo Reply with exactly one sentence: The quick brown fox.
  • El backend soporta proveedor none local, pero el admin meta no lo anuncia.
  • Si llamas por API sin classifier_*, hereda provider/model del moderator.
Step 4. Treatment Groups scope: experimental.groups
chatroom_context = "Spanish Telegram chat" redirect_url = "" groups = 9 cells (3x3) not_incivil / mix / incivil x not_like_minded / mix / like_minded

El unico default fuerte aqui es ecological_validity_criteria, que arranca con una descripcion larga del estilo "dialogic + mix de acciones + mensajes cortos + tono reddit-like". El preset de frontend carga una matriz 3x3 completa con incivilidad y alineacion independientes.

  • El criterio ecologico es global al experimento.
  • El criterio interno es por grupo y es obligatorio para guardar.
  • El builder 2x2 y el preset 3x3 son ayudas de frontend, no reglas del backend.
  • Si activas news_article, ese grupo puede llevar un seed article.
Step 5. Participant Tokens scope: enrolment
participants per group UI default = 10 tokens cryptographically random single use
  • El numero por grupo es un estado del wizard, no parte del config JSON persistido.
  • Guardar requiere que existan tokens y que sus grupos coincidan exactamente con los grupos del experimento.
Step 6. Review & Save save
  • POST /admin/config valida, guarda config y siembra tokens en DB.
  • La UI antigua y parte del README hablan de "lock", pero el codigo actual tambien soporta editar con PUT.
  • Despues el dashboard permite activar, pausar, clonar, resetear sesiones, borrar y exportar.

Default que mas impacta el comportamiento

  • ecological_validity_criteria ya empuja a mezcla de tipos de accion y mensajes cortos.
  • messages_per_minute = 6 hace un experimento bastante activo para 5 minutos.
  • performer_memory_size = 3 es la principal barrera anti-repeticion en el Performer.

Diferencias wizard vs fallback backend

  • performer_max_tokens: wizard 256, fallback backend 512 si falta.
  • action_window_size: wizard 5, fallback backend 10 si falta.
  • classifier_prompt_template: no se edita en UI, pero backend exige uno no vacio y usa un default de fichero.

Providers sugeridos en meta

  • anthropic, gemini, huggingface, mistral, konstanz, bsc.
  • El doc de BSC recomienda usar bsc/incivility como Performer, no como Director.
  • El wizard avisa si un provider no tolera temperature y top_p a la vez.

Workflow runtime

Aqui esta el flujo completo, desde el token del participante hasta el cierre de la sesion. Los botones cambian el detalle del paso activo.

1. Token -> pending session

El frontend llama POST /session/start con el token. El backend lo consume de forma atomica, resuelve treatment_group y experiment_id, comprueba ventana temporal y pausa, y reserva una sesion pendiente.

Input token de un solo uso
Output session_id
Persistencia sessions.status = pending
  • Si el experimento no esta disponible, el token se desmarca como usado para que la persona pueda reintentar.
  • El nombre visible del participante no se envia aqui.

2. WebSocket -> create/reconstruct session

Al conectar al WebSocket, el sistema recupera la pending session o reconstruye una sesion activa desde DB/Redis si hubo reconexion o crash recovery.

Carga config del experimento + mensajes previos + agent blocks
Inicializa SimulationSession, LLM managers, features, orchestrator
Arranque seed features + clock loop + Redis pub/sub
  • Los agentes se crean a partir de agent_names y agent_personas.
  • Si la feature news_article esta activa, mete un mensaje [news] al empezar.
  • El clock loop es secuencial; solo un turno de agente a la vez.

3. Director Update

Si ya hubo actividad previa y no fue un like, el Director actualiza el perfil del ultimo performer que actuo. Ese perfil es un resumen emergente de posiciones, estilo e interacciones.

Mira ultimo performer + su perfil previo + su ultima accion
Devuelve performer_profile_update
Editable no desde UI

4. Director Evaluate

Evalua dos ejes: validez interna del tratamiento y validez ecologica del chat. Durante el warm-up corre en cada turno; despues corre cada evaluate_interval turnos productivos.

Contexto ultimos evaluate_interval mensajes
Mira tambien action counts + participation counts
Devuelve dos summaries textuales
  • El humano se excluye de la presion por equidad de participacion.
  • Si el Director "espera" al humano, el contador de evaluate se revierte.

5. Director Action

A partir de evaluaciones + perfiles + chat log, el Director elige next_performer, action_type, target y una instruccion O/M/D: objective, motivation y directive.

Action types message, reply, @mention, like
Ventana ultimos action_window_size mensajes
Caso especial si elige al humano, el turno pasa a wait
  • Si el Director pide self-mention, el sistema la degrada a message.
  • Si elige un agente invalido, cae a uno aleatorio valido.
  • Un like duplicado tambien se convierte en wait.

6. Performer

El Performer no ve el chat completo. Solo recibe contexto del chatroom, su perfil, sus mensajes recientes, la O/M/D del Director y el target relevante.

Memoria performer_memory_size
Objetivo producir solo el mensaje visible
Reintentos hasta 3 con Moderator
  • Para message con target_user, el sistema busca el ultimo mensaje de esa persona.
  • La salida aun puede traer ruido, por eso siempre pasa por Moderator.

7. Moderator

Recorta razonamiento, prefijos, markdown o basura del Performer. Si no detecta contenido, devuelve NO_CONTENT y se reintenta el ciclo Performer -> Moderator.

Input salida cruda del Performer
Output texto limpio o NO_CONTENT
Si falla 3 veces el turno se oculta como wait

8. Classifier

Despues de moderar el mensaje, corre un clasificador extra para etiquetar incivility, like-mindedness e inferencia de stance del participante.

Input historial del participante + mensaje del agente
Output is_incivil, is_like_minded, stance, rationale
Configurable modelo y prompt template via API
  • Si el classifier falla, el mensaje sigue existiendo; solo se pierden labels.
  • Las columnas quedan en messages para export y evaluacion.

9. Fin de sesion

Si la sesion expira por tiempo, publica un evento session_end con redirect_url, cierra WebSocket y persiste el cierre. El frontend redirige o muestra la thank-you screen.

Trigger duration_minutes
Frontend limpia sessionId de localStorage
Post-run session report HTML + CSV + dashboard exports

Condiciones y reglas fijas

Tick loop: cada 1 segundo.
Tipado simulado: 7 chars/seg, minimo 0.5s, maximo 8s.
Secuencial: un lock evita turnos concurrentes para que el Director vea todo en orden.
Broadcast: Redis pub/sub entrega a WebSocket; DB es la fuente autoritativa.
Bloqueo por report: no borra mensajes viejos; solo oculta futuros del agente bloqueado.
El pipeline real tiene 6 llamadas LLM, no 5. README enumera 5 porque no cuenta explicitamente el classifier post-moderation.

Performer

El runtime usa el template unificado backend/agents/STAGE/prompts/performer_prompt.md, no el fichero legacy en prompts/system/performer_prompt.md.

Inputs: chatroom context + agent profile + recent messages + O/M/D + target Action blocks condicionales Debe devolver solo el mensaje
Ver template crudo 
You are a 'Performer' in a social-scientific experiment simulating a realistic online chatroom. Read the instructions below, which will guide you to write a short message for character. Follow the instructions exactly. Output ONLY the message.

## About the Chatroom:

{CHATROOM_CONTEXT}

## Your Participation So Far:
{AGENT_PROFILE}

## Your Most Recent Messages:
{RECENT_MESSAGES}

You want to: {OBJECTIVE}
This matters to you because: {MOTIVATION}
Your message must be: {DIRECTIVE}

{#ACTION_TYPE: message}
Post a new message to the chatroom. You are not responding to anyone in particular.
{/ACTION_TYPE}

{#ACTION_TYPE: message_targeted}
Post a message in response to {TARGET_USER}'s most recent message:
> {TARGET_MESSAGE}
{/ACTION_TYPE}

{#ACTION_TYPE: reply}
Reply to this earlier message. The reader will see it quoted above your reply:
> {TARGET_MESSAGE}
{/ACTION_TYPE}

{#ACTION_TYPE: @mention}
Post a message directed at @{TARGET_USER}. Do not include the @mention - it is added automatically.
{/ACTION_TYPE}

Moderator

El runtime usa backend/agents/STAGE/prompts/moderator_prompt.md. Es un extractor minimalista: quita explicaciones, markdown y prefijos del Performer.

Salida esperada: solo chatroom message Sentinela: NO_CONTENT No editable desde UI
Ver template crudo 
{#SYSTEM}
You must extract the chatroom message from the text below. The text should contain a chatroom message, but may include extra content (e.g., reasoning, commentary, formatting, etc.).

Strip all extra content. Return ONLY the chatroom message itself. No quotation marks. Preserve the original language exactly.

If no identifiable chatroom message is present, output exactly: NO_CONTENT
{/SYSTEM}

{#USER}
{PERFORMER_OUTPUT}
{/USER}

Prompt plumbing

  • prompt_renderer.py conserva bloques {#SYSTEM} o {#USER} segun el modo.
  • Para Performer, tambien resuelve bloques {#ACTION_TYPE: ...}.
  • Los placeholders se rellenan con chatroom_context, summaries, perfiles y targets.

Que prompt puedes tocar sin codigo

  • Directamente en wizard: ninguno de los templates base.
  • Indirectamente: chatroom_context, criterios, personas, nombres y modelos si alteran la entrada al prompt.
  • Via API: classifier_prompt_template.

Prompts legacy

  • Existen backend/agents/STAGE/prompts/system/director_prompt.md, system/performer_prompt.md y system/moderator_prompt.md.
  • No aparecen cableados en director.py, performer.py o moderator.py.
  • Conviene tratarlos como material historico, no como fuente de verdad del runtime.
