WP5 / STAGElab

Variables que puedes cambiar y workflow que sigue antes de contestar

Esta guia esta enfocada en dos preguntas operativas: que variables puedes tocar en tu experimento y que hace exactamente el sistema antes de producir una respuesta visible. El objetivo no es tanto explicar arquitectura general como dejar claro que controla cada capa, que decide cada paso y que cosas no puede hacer cada etapa del pipeline.

Lectura rapida

Lo importante para operar el sistema es esto: el experimento se guarda como JSON por experimento, pero la respuesta visible no la decide un unico prompt. Antes de salir al chat pasa por gating, evaluacion, seleccion de accion, redaccion, moderacion y clasificacion, y cada paso tiene limites distintos.

Que ve cada rol

Los prompts no ven lo mismo

  • Director: ve chat log anonimizado, perfiles acumulados y evaluaciones de validez.
  • Performer: no ve el chat log completo; solo perfil, sus mensajes recientes, O/M/D y target.
  • Moderator: solo limpia la salida del Performer.
  • Classifier: clasifica el mensaje final del agente tras moderacion.

Hay varios puntos menos obvios que la UI no explica

Privacidad y anonimizado

El participante real no se expone al backend con su nombre visible. La sesion lo trata como participant y el orquestador lo reetiqueta junto con los agentes como Performer 1, Performer 2, etc. El mapeo se baraja con random_seed y se mantiene estable toda la sesion.

Wizard de configuracion

La UI de administracion vive en Next.js y divide la configuracion en 6 pasos. Los defaults del wizard son frontend-owned, mientras que las opciones disponibles de proveedores y modelos se piden al backend por /admin/meta. Si buscas solo las palancas reales del experimento, la seccion siguiente de Variables es la vista mas directa.

Punto de consistencia

README, StepReview y el modal de guardado siguen hablando de configuracion "locked" o inmutable. La implementacion actual ya soporta edicion de experimentos existentes por PUT /admin/config/{experiment_id}. Si quieres documentacion estrictamente fiel al codigo actual, conviene mencionar ambos hechos.

Experiment Identity

1

4

Variables que puedes cambiar

Esta tabla separa variables de sesion, pipeline LLM, tratamiento y ajustes ocultos. Usa los filtros para ver solo las que te interesan.

Como leer esta matriz

  • Donde se cambia: distingue entre wizard, API/JSON o codigo.
  • Impacta: resume si cambia el ritmo, la seleccion de acciones, el contenido o el postproceso.
  • Limite: aclara hasta donde llega ese control y donde deja de tener efecto.
Variable Donde se cambia Impacta Limite real
session_duration_minutes Wizard, simulation Duracion de la sesion. No cambia el contenido; solo cuando termina el experimento y cuando se envia session_end.
num_agents y agent_names Wizard, simulation Numero de performers disponibles y nombres visibles. No define por si mismo personalidad, tono ni stance.
agent_personas Wizard / JSON En teoria personalidad por agente. En el runtime actual no aparece conectada a los prompts STAGE principales.
random_seed Wizard, simulation Shuffle anonimo y partes aleatorias del pacing. No fija un guion; solo hace mas reproducible la aleatoriedad compatible con el runtime.
messages_per_minute Wizard, simulation Probabilidad de que se dispare un turno por tick. No garantiza ritmo exacto porque la latencia LLM y los waits tambien limitan la salida real.
evaluate_interval Wizard, simulation Cada cuanto el Director reevalua validez. No cambia directamente el texto; cambia cuan a menudo el Director corrige deriva.
action_window_size Wizard, simulation Cuantos mensajes recientes ve el Director en Action. Afecta contexto de decision, no el estilo literal del mensaje final.
performer_memory_size Wizard, simulation Cuantos mensajes propios recientes ve el Performer. Ayuda a evitar repeticion; no le da acceso al chat log completo.
chatroom_context Wizard, experimental Marco global del escenario. Orienta a todos los roles, pero no sustituye criterios de tratamiento.
ecological_validity_criteria Wizard, experimental Naturalidad, estilo y dinamica que debe preservar el Director. No dicta una frase exacta; condiciona Evaluate y luego Action.
groups.*.internal_validity_criteria Wizard, por grupo Manipulacion experimental real. Es la variable mas fuerte para cambiar el comportamiento agregado del chat, pero sigue siendo una guia de control, no un script determinista.
groups.*.features y seed Wizard, por grupo Semilla inicial y gating de agentes. Solo hay dos features activas en registry: news_article y gate_until_user_post.
redirect_url Wizard, dentro de experimental Destino tras session_end. No afecta ninguna decision del pipeline conversacional.
director_* / performer_* / moderator_* / classifier_* Wizard, simulation Modelo, temperatura, top_p y max_tokens por rol. Si cambias modelo o sampling puedes cambiar mucho el comportamiento, pero cada rol sigue limitado a su propio prompt y responsabilidad.
classifier_prompt_template API / JSON / codigo Prompt user del classifier. No se expone en el wizard; solo cambia como se etiquetan mensajes, no el texto generado.
Prompts Markdown base Codigo Instrucciones estructurales de cada rol. Esto ya no es parametrizacion de experimento: cambia el comportamiento global de la plataforma.

Si quieres cambiar de verdad el experimento

Las tres palancas con mas impacto

  • internal_validity_criteria: cambia la manipulacion experimental.
  • ecological_validity_criteria: cambia el estilo y naturalidad que se intenta preservar.
  • role models: cambia la calidad, coste y tendencia de cada etapa del pipeline.

Variables utiles pero menos transformadoras

Cambian la operativa mas que la semantica

  • session_duration_minutes, tokens y redirect.
  • messages_per_minute y evaluate_interval.
  • Nombres de agentes y numero de participantes por grupo.

Workflow antes de cada respuesta

Si lo que quieres saber es “que pasa antes de que el sistema conteste”, mira primero la pestaña de turno. La pestaña de arranque de sesion queda como apoyo para entender el contexto.

Resumen en una linea

El texto visible lo redacta el Performer. Antes de eso, el sistema decide si hay turno, resume contexto, evalua validez y escoge accion; despues del Performer, solo se limpia, etiqueta y persiste.

Tick base: 1.0s Pacing: messages_per_minute / 60 Typing speed: 7 chars/s Delay typing: 0.5s - 8.0s Retries Performer: 3

Detalles de control que afectan el comportamiento

  • Si el Director selecciona al humano, el turno se convierte en wait y no cuenta como turno productivo.
  • Los like no llaman al Performer ni al Moderator.
  • Durante warm-up, Evaluate corre en cada turno; luego pasa a cada evaluate_interval turnos productivos.
  • La clasificacion de incivility / like-mindedness ocurre despues de la moderacion, sobre el mensaje final del agente.

Prompts por defecto

Los prompts base viven en Markdown dentro de backend/agents/STAGE/prompts. Director, Performer y Moderator usan plantillas unificadas con bloques {#SYSTEM}, {#USER} y, en el Performer, {#ACTION_TYPE: ...}.

Dos cosas importantes sobre los prompts

  • Los textos base de prompt estan escritos en ingles, aunque el experimento por defecto precargue contexto y mensajes en espanol.
  • La mayor parte de la personalizacion no cambia el markdown base: entra por placeholders como {CHATROOM_CONTEXT}, {INTERNAL_VALIDITY_CRITERIA} o {OBJECTIVE}.

Director - Update Performer Profile

Reescribe el perfil acumulado del ultimo performer que actuo. Sirve para que el Director mantenga memoria condensada de posiciones, interacciones y estilo.

Variables inyectadas

{CHATROOM_CONTEXT} {LAST_AGENT} {LAST_AGENT_PROFILE} {LAST_ACTION}

Contrato de salida

JSON con una sola clave: performer_profile_update.

Que se puede cambiar sin tocar el prompt

Solo el chatroom_context y, de forma emergente, el contenido de la conversacion.

Ver prompt base 
# Director - Update Performer Profile

You are the 'Director' in a social-scientific experiment simulating a realistic online chatroom. Your role is strictly behind the scenes. In this step, you must update the profile of the performer that acted most recently.

## Chatroom Context

Here is the chatroom context, as described by the researcher for this experiment:

`{CHATROOM_CONTEXT}`

{#SYSTEM}
The name of the last-acting performer, their current profile, and their most recent action will be provided in the user message below.
{/SYSTEM}

## Your Task

Read the performer's most recent action and update their profile accordingly.

A performer profile is a running characterisation of how that performer has behaved so far: positions taken, significant interactions, and communication style. Profiles describe what the performer has *done*, not who they *are*. They are the emergent record of each performer's participation. Each update should be a complete revision that replaces the previous profile, retaining important earlier information while incorporating what is new. Prioritise the performer's most recent positions and most significant interactions. Keep the profile concise (1-5 sentences).

## Output Format

Respond with a JSON object using exactly this structure:

```json
{
  "performer_profile_update": "Updated profile for the last-acting performer (1-5 sentences)."
}
```

# Director - Design Action

You are the 'Director' in a social-scientific experiment. Your purpose is to ensure the simulated chatroom achieves two goals: **internal validity** and **ecological validity**. You pursue these goals by deciding which performer should act next and shaping their action through structured instructions.

### Step 1: Identify the Priority
Read the validity evaluations.

### Step 2: Select a Performer
Read the performer profiles and participation counts.

### Step 3: Select an Action

- `message`
- `reply`
- `@mention`
- `like`

Controles avanzados y limites

Esta seccion ya no resume todas las variables, sino las zonas menos visibles: API, codigo y diferencias entre lo que parece configurable y lo que realmente afecta al runtime.

Desde el wizard

Casi todo lo operativo del experimento

  • Identidad: experiment_id, descripcion, fechas, redirect.
  • Sesion: duracion, numero de agentes, nombres, pacing, seed, evaluate interval, action window, performer memory.
  • LLMs: provider, model, temperature, top_p y max_tokens para Director, Performer, Moderator y Classifier.
  • Tratamientos: contexto, ecological validity, grupos, internal validity, features y seed article.
  • Tokens: cantidad por grupo y CSV.

Por API o JSON

Campos soportados que la UI no expone del todo

  • simulation.classifier_prompt_template si quieres cambiar la plantilla user del classifier sin tocar codigo.
  • PUT /admin/config/{experiment_id} si quieres actualizar un experimento ya existente.
  • Cualquier payload que monte directamente { simulation, experimental } mientras pase la validacion del backend.
Ojo: si guardas por el wizard, siempre se enviaran los defaults frontend ya resueltos.

Solo tocando codigo

Comportamientos globales y defaults estructurales

  • Los prompts markdown de Director, Performer y Moderator.
  • El prompt system del Classifier y el user template base por defecto.
  • El registro de features disponibles y sus labels.
  • El registro de providers/modelos sugeridos.
  • Constantes runtime como MAX_PERFORMER_RETRIES o la velocidad de typing.
  • El preset 3x3 que precarga el wizard al abrir un experimento nuevo.

Puntos a vigilar

Cosas que parecen una cosa pero hoy se comportan de otra

  • agent_personas: existe en config y UI, pero no aparece usada en la cadena de prompts del runtime actual.
  • Classifier: configurable y testeable, pero el wizard no bloquea el guardado si su test no se ha ejecutado.
  • La copia textual de "Save & Lock" no refleja el soporte actual de edicion por PUT.
  • Para Director, Performer y Moderator, el runtime carga las plantillas unificadas de la raiz de prompts/; los archivos separados en prompts/system y prompts/user parecen legacy.
